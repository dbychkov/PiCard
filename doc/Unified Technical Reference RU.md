# PiCard Прошивка и Аппаратная часть — Единое Техническое Руководство

> Данный документ объединяет анализ исходного кода прошивки со схемотехнической документацией (ZX\_PICARD\_V3) и является единым авторитетным справочником по плате расширения PiCard RP2350B для ZX Spectrum.

---

## Общий Обзор

### Краткое Описание

PiCard — это **плата расширения, совместимая с шиной NemoBus** ZX Spectrum, которая заменяет целый ряд классических периферийных микросхем одним модулем **WeAct Studio RP2350B** (микроконтроллер Raspberry Pi RP2350 + 8 МБ встроенной PSRAM). Прошивка эмулирует сопроцессорную карту **General Sound (GS)**, двухканальную плату **Turbo Sound** (два чипа AY), SD-карточный интерфейс, совместимый с **DivMMC**, интерфейс **Z-Controller**, адаптер **клавиатуры PS/2** и интерфейс **джойстика Kempston**. Все источники звука смешиваются в реальном времени и подаются на **ЦАП TDA1387T** (U4 на схеме) через выходной каскад на операционном усилителе MCP602 в стереофонический разъём PJ325.

Прошивка написана на **C/C++** под инструментальную цепочку Arduino-RP2350 (в основе — Pico SDK).

---

### Аппаратная Архитектура

RP2350B разогнан до **351 МГц** (напряжение питания ядра повышено до 1,60 В через `vreg_set_voltage(VREG_VOLTAGE_1_60)`), что обеспечивает достаточный запас производительности для одновременного выполнения программного Z80, обслуживания шины ZX на частоте 3,5 МГц в режиме опроса и синтеза многоканального звука каждые 9 мкс.

Плата работает в двух изолированных доменах питания:

| Домен | Напряжение | Ключевые компоненты |
|-------|-----------|---------------------|
| ZX NemoBus | +5 В КМОП | Краевые разъёмы J1/X1A + J2/X1B, 74HC245 U1, 74HC04 U2, транзисторы BC817 Q1–Q4 |
| Цифровая логика RP2350B | +3,3 В | Модуль WeAct, PCF8575 U3, TDA1387T U4, UART1, SPI1 (SD-карта) |

**LDO-стабилизатор AMS1117-3.3** (U6) формирует шину +3,3 В из шины +5 В. Питание +5 В поступает с краевого разъёма Spectrum через паяные перемычки JP\_5V0/JP\_5V1/JP\_5V2.

#### Интерфейс NemoBus

Линии шины Spectrum подключены непосредственно к банкам GPIO RP2350:

| Сигнал | GPIO | Направление | Путь на схеме |
|--------|------|-------------|---------------|
| A0–A15 | 8–23 | Вход | Через резисторные сборки RP1/RP2 (10 кОм) на ЗЗ-ограничитель |
| D0–D7 | 24–31 | Двунаправленный | Через 74HC245 U1 (сторона A = 3,3 В, сторона B = 5 В NemoBus) |
| /M1 | 3 | Вход | RP2 ? GPIO 3 (последовательно 10 кОм) |
| /WR | 4 | Вход | RP2 ? GPIO 4 |
| /RD | 5 | Вход | RP1 ? GPIO 5 (также вывод JMP для PIO1) |
| /IORQ | 6 | Вход | RP1 ? GPIO 6 |
| /MREQ | 7 | Вход | RP1 ? GPIO 7 |
| DATA\_CTRL | 2 | Выход | GPIO 2 ? база BC817 Q1 ? коллектор Q1 управляет DIR/CE 74HC245 (5 В) |
| ROM\_BLK | 1 | Выход | GPIO 1 ? BC817 Q4 ? буфер 74HC04 U2 ? цепь BLK краевого разъёма |
| /NMI | 35 | Открытый сток | GPIO 35 ? BC817 Q3 ? подтяжка R30 (300 Ом) к +5 В ? NemoBus /NMI |
| /RESET | 47 | Открытый сток | GPIO 47 ? BC817 Q2 ? NemoBus /RESET |

`PIN_DATA_CTRL` (GPIO 2) управляется выводом `side-set` PIO, что позволяет аппаратно включать выход 74HC245 лишь на те наносекунды, которые необходимы для выставления данных на шину. `PIN_ROM_BLK` (GPIO 1) управляет буфером блокировки ПЗУ, позволяя RP2350 подставлять собственные образы ROM (например, ПЗУ DivMMC) поверх внутреннего ПЗУ Spectrum.

#### Подключение Периферии

| Периферия | Интерфейс | Выводы | Физический компонент |
|-----------|-----------|--------|----------------------|
| SD-карта | SPI1 (10 МГц) | RX=40, CS=41, CLK=42, TX=43 | Разъём Micro-SD J\_SD1 + 6-контактный разъём J7 |
| ЦАП I?S | PIO2 | DATA=44, BCLK=45, LRCK=46 | TDA1387T U4, выводы 3/1/2 |
| Расширитель джойстика PCF8575 | I?C0 (400 кГц) | SDA=32, SCL=33 | PCF8575DBR U3, выводы 23/22 |
| Клавиатура PS/2 | PIO0 | CLK=38, DATA=39 | Mini-DIN-6 J6, выводы 5/1 через R12/R13 (1 кОм) |
| UART (протокол ZX UNO) | UART1 (115200) | TX=36, RX=37, CTS=34 | Вспомогательный разъём J8 / совместно с PS/2 порт 2 |
| PSRAM | QMI/XIP CS1 | GPIO 0 | APS6404L на модуле WeAct RP2350B |

## Подробный Анализ Эмулируемых Устройств

### Карта General Sound (GS)

Карта GS — это автономный сопроцессорный модуль на базе Z80. Прошивка эмулирует всю карту программно, используя встроенную PSRAM для хранения банкированной ОЗУ и ЦАП TDA1387T для вывода звука.

#### Эмуляция CPU Z80

Z80 для GS эмулируется портируемым программным Z80-ядром (`src/GS/z80/Z80.c`, на основе эмулятора Marcel de Kogel), создаваемым как единый глобальный объект `Z80 cpu` на **Ядре 0**. `GS.cpp` предоставляет четыре обязательных коллбэка:

- **`RdZ80(address)`** — чтение памяти; маршрутизирует к ПЗУ, SRAM или QMI PSRAM в зависимости от адреса и состояния банкирования. Декорирован `__time_critical_func` для принудительного размещения в SRAM (не в XIP-флэш), что исключает задержки промахов кэша при выполнении Z80.
- **`WrZ80(address, value)`** — запись в память; обновляет соответствующий банк ОЗУ.
- **`InZ80(port)`** — чтение ввода-вывода Z80; открывает регистры межпроцессорного почтового ящика для Z80 GS.
- **`OutZ80(port, value)`** — запись ввода-вывода Z80; обрабатывает разбивку на страницы (MPAG), громкость каналов и запись в почтовый ящик.

Z80 тактируется в основном цикле Ядра 0 с использованием времени, оставшегося до следующего аудиотика в 9 мкс:

```c
int32_t dt = int_tick - tick_time;
ExecZ80(&cpu, dt * 12);  // ~12 тактов Z80 на мкс при эмулируемых 12 МГц
```

На каждом аудиотике в 9 мкс подаётся прерывание IRQ и захватывается последний стереосемпл GS:

```c
IntZ80(&cpu, INT_IRQ);
snd_out_GS = GS_get_sound_LR_sample();
```

NMI и сброс инициируются записью в регистр `GSCTR` (порт `0x33`) со стороны Spectrum: биты `0x40` (NMI) и `0x80` (сброс) обрабатываются в цикле Ядра 0.

#### Банкирование Памяти — На Основе Встроенной PSRAM

64-страничное банкированное адресное пространство карты GS хранится в чипе **PSRAM на 8 МБ**, являющемся частью модуля WeAct RP2350B (APS6404L, подключён к RP2350 через QMI). Прошивка инициализирует интерфейс QMI через `psram_init(SFE_RP2350_XIP_CSI_PIN)`:

```c
// Делитель QMI установлен так, чтобы тактовая частота PSRAM ? 166 МГц (максимум APS6404L)
// При частоте системы 351 МГц, делитель = 3 ? частота PSRAM ? 117 МГц
gpio_set_function(cs_pin, GPIO_FUNC_XIP_CS1);
qmi_hw->m[1].timing = ... rxdelay=3, divisor=3 ...
xip_ctrl_hw->ctrl |= XIP_CTRL_WRITABLE_M1_BITS;  // разрешить запись в PSRAM через XIP
```

Большие банкированные массивы размещаются в PSRAM с помощью атрибута секции `PSRAM`:

```c
static uint8_t RAM_BANK1[62][0x4000] PSRAM;  // страницы 2–63, 62?16 КБ = 992 КБ
static uint8_t RAM_BANK2[62][0x4000] PSRAM;  // второй набор, ещё 992 КБ
```

Страницы 0 и 1 используют быструю локальную SRAM RP2350 (`RAM_BANK1_1[2]`, `RAM_BANK1_2[2]`), чтобы наиболее часто используемые страницы не несли задержек PSRAM.

Четыре указателя слотов (`ram_slot1`–`ram_slot3`) обновляются командой `OutZ80(port=0x00, value=page)` для указания на текущую страницу PSRAM:

| Слот | Диапазон адресов | Содержимое по умолчанию |
|------|-----------------|------------------------|
| slot0 | `0x0000–0x3FFF` | ПЗУ GS (`gs105b_rom[]`, во флэш) |
| slot1 | `0x4000–0x7FFF` | Фиксированная страница SRAM (RAM_BANK1_2[1]) |
| slot2 | `0x8000–0xBFFF` | Банкированная (страница PSRAM `&RAM_BANK1[page-2]`) |
| slot3 | `0xC000–0xFFFF` | Банкированная (страница PSRAM `&RAM_BANK2[page-2]`) |

#### Захват PCM-Каналов

Хитрый трюк внутри `RdZ80()` автоматически захватывает данные PCM-семплов: когда Z80 GS читает адреса `0x6000–0x7FFF`, биты A8 и A9 адреса молча определяют, в какой из четырёх аудиоканалов направляется байт:

| A9 | A8 | Захватываемый канал |
|----|----|---------------------|
| 0  | 0  | `channel1` (0x6000–0x60FF) |
| 0  | 1  | `channel2` (0x6100–0x61FF) |
| 1  | 0  | `channel3` (0x6200–0x62FF) |
| 1  | 1  | `channel4` (0x6300–0x63FF) |

Это происходит пассивно во время обычных операций чтения памяти Z80 GS — без прерываний и DMA.

#### Интерфейс Ввода-Вывода со Стороны ZX

Spectrum взаимодействует с GS через три порта ввода-вывода, декодируемых в цикле опроса шины на Ядре 1:

| Порт | Направление | Функция |
|------|-------------|---------|
| `0xB3` | Запись | ZX записывает байт данных в GS (`GSDAT`); устанавливает `DataBIT` в `GSSTAT` |
| `0xB3` | Чтение | ZX читает последний ответ GS (`ZXDATWR`); сбрасывает `DataBIT` |
| `0xBB` | Запись | ZX посылает байт команды в GS (`GSCOM`); устанавливает `CommandBIT` в `GSSTAT` |
| `0xBB` | Чтение | ZX читает байт статуса GS (`GSSTAT`) |
| `0x33` | Запись | ZX записывает управление GS (`GSCTR`): бит7=сброс, бит6=NMI |

Эти порты записывают в переменные `volatile uint8_t`, разделяемые между Ядром 0 (коллбэки Z80 GS) и Ядром 1 (цикл шины). Однобайтовые `volatile`-записи по своей природе являются однотактовыми и безопасны без дополнительных примитивов синхронизации на ARMv8-M.

#### Аппаратный Путь Звука — От Z80 GS до TDA1387T

```
GS Z80 (Ядро 0, программно) ? channel1–channel4 (uint8_t, PCM 8 бит без знака)
  ? GS_get_sound_LR_sample()
       применяет громкость (порт 0x06–0x09) и стерео-перекрёстную связь:
       snd.L = ch3*vol3 + ch4*vol4 + ch1*(vol1/3) + ch2*(vol2/3)
       snd.R = ch1*vol1 + ch2*vol2 + ch3*(vol3/3) + ch4*(vol4/3)
  ? snd_out_GS (volatile sound_LR, 2?int16_t)
  ? mix_sound_out() суммирует с TS, Beeper, COVOX, SND_DRIVE
  ? i2s_out(L_clipped, R_clipped)
       упаковывает в static uint32_t i2s_data = (L<<16)|R
  ? DMA-канал dma_i2s читает i2s_data (без инкремента) ? TX FIFO PIO2
       темп задаётся сигналом DREQ PIO2 на 44100 Гц
  ? Автомат состояний PIO2 (audio_i2s_program) сериализует биты ? GPIO 44 (DATA)
       side-set переключает GPIO 45 (BCLK) и GPIO 46 (LRCK)
  ? TDA1387T U4 (ведомый I?S): DATA вывод 3, SCLK вывод 1, LRCK вывод 2
  ? Аналоговые выходы TDA1387T A_LEFT / A_RIGHT
  ? Буфер MCP602 U5A/U5B + антиалиасинговый фильтр 5,5 кГц
  ? Стереоразъём PJ325 J5 3,5 мм (Tip=L, Ring=R)
```

### Turbo Sound — Два Чипа AY-3-8910 в Одном Потоке I?S

Turbo Sound предоставляет Spectrum **два чипа AY-3-8910** по одному адресу ввода-вывода. Выделенного аппаратного обеспечения для чипов AY в схеме нет; весь синтез выполняется программно функцией `TS::step()` на **Ядре 0**, а выход смешивается в тот же стереосемпл `snd_out_TS`, который поступает на TDA1387T.

#### Декодирование Портов

Оба порта — выбора регистра AY (`0xFFFD`) и записи данных (`0xBFFD`) — декодируются в цикле шины Ядра 1 с помощью маскированных сравнений, соответствующих оригинальному соглашению AY-шины, где A14 и A15 выбирают функцию чипа, а A1 — сам чип:

```c
// Фиксация / выбор регистра
if ((addr16 & 0xC003) == (0xFFFC & 0xC003)) ts.select_reg(d8);
// Запись данных
if ((addr16 & 0xC003) == (0xBFFC & 0xC003)) ts.set_reg(d8);
// Чтение данных
if ((addr16 & 0xC003) == (0xFFFC & 0xC003)) put_dataZ80(ts.get_reg());
```

Выбор чипа осуществляется сигнальными значениями, записываемыми в порт фиксации (`0xFF` ? чип 0, `0xFE` ? чип 1).

#### Движок Синтеза (`TS::step()`) — Вызывается на Ядре 0 с частотой 111 кГц

`step()` вызывается один раз за каждый аудиотик в 9 мкс. Каждый вызов обрабатывает **оба чипа** в цикле. Для каждого чипа:

1. **Генераторы тонов** (каналы A, B, C): счётчик на канал инкрементируется на `DELTA=2`; при превышении 12-битного регистра периода бит выхода переключается. Интерполяция между семплами (остатки `dA`, `dB`, `dC`) уменьшает высокочастотные алиасинг-артефакты.
2. **Генератор шума**: 32-битный LFSR (`get_random()`) продвигается со скоростью, заданной R6 (регистр 6). Каналы с включённым шумом выполняют XOR выхода с текущим битом шума.
3. **Генератор огибающей**: 32-элементная таблица форм `Envelopes[16][32]` (предварительно вычисленная при запуске из `Envelopes_const` и `ampls_AY`) тактируется со скоростью R11/R12. Циклические формы перезапускаются с индекса 0; удерживаемые формы замирают на шаге 31.
4. **Таблица амплитуд**: уровень выхода читается из `ampls_AY[16]` (логарифмическая шкала, значения до 21845).
5. **Стерео-микширование**: Чип 0 направляет A+B+?B вправо, C+B+?B влево. Чип 1 добавляется аналогично. Оба чипа накапливаются в `sample_out`.

Результат `sound_LR` сохраняется в `snd_out_TS` и суммируется в глобальном микшере в `mix_sound_out()`.

#### Аппаратный Путь Звука

```
TS::step() (Ядро 0, каждые 9 мкс) ? snd_out_TS (sound_LR)
  ? mix_sound_out() ? i2s_out() ? DMA ? PIO2 ? TDA1387T U4 ? MCP602 ? J5
```

Конечный путь идентичен GS — оба эмулируемых устройства разделяют единственный ЦАП TDA1387T I?S и аналоговый выходной каскад MCP602. TDA1387T получает одно объединённое 32-битное слово I?S на семпл, содержащее полностью смешанные левый и правый каналы от всех одновременно активных источников звука.

### DivMMC — Интерфейс SD-Карты через SPI1 и Перехват M1 шины NemoBus

DivMMC — это SD-карточный интерфейс с отображением в память, который перехватывает определённые адреса выборки опкодов Z80 (M1) для активации окна ПЗУ/ОЗУ размером 16 КБ в нижней части адресного пространства Spectrum.

#### Физический Интерфейс SD-Карты

SD-карта подключается к **SPI1** (аппаратный SPI-периферийный модуль) через:

| Сигнал SPI1 | GPIO | Цепь на схеме | Разъём |
|-------------|------|---------------|--------|
| MISO (RX) | 40 | SD\_MISO | Micro-SD J\_SD1 + J7 вывод 4 |
| CS0 | 41 | SD\_CS0 | J\_SD1 + J7 вывод 1 |
| CLK | 42 | SD\_CLK | J\_SD1 + J7 вывод 3 |
| MOSI (TX) | 43 | SD\_MOSI | J\_SD1 + J7 вывод 2 |

SPI1 инициализируется на **10 МГц** в `ZX_bus_task()` на Ядре 1. Отдельные байты передаются путём прямого обращения к аппаратному регистру данных SPI для минимальной задержки на один байт без накладных расходов SDK:

```c
static inline uint8_t READ_SD_BYTE() {
    uint8_t data = spi_get_hw(SD_SPI)->dr;
    spi_get_hw(SD_SPI)->dr = 0xFF;  // предзагрузка следующего фиктивного байта
    return data;
}

static inline void WRITE_SD_BYTE(uint8_t data) {
    volatile uint8_t dummy = spi_get_hw(SD_SPI)->dr;  // сброс RX
    spi_get_hw(SD_SPI)->dr = data;
}
```

Прошивка использует **два физических разъёма SD-карты** (J\_SD1 и J7), подключённых параллельно к одной шине SPI1 и одной линии CS0 (GPIO 41). Одновременно можно вставить только одну карту.

#### Окно Памяти

Когда DivMMC активен (`divMMC_ON` или `divMMC_ON_PORT`):

| Адрес | Содержимое | Физический источник |
|-------|-----------|---------------------|
| `0x0000–0x1FFF` | ПЗУ DivMMC (`ROM_DIVMMC[]`, 8 КБ ESXDOS) | Массив C во флэш |
| `0x2000–0x3FFF` | Выбранная 8 КБ-страница ОЗУ из `divMMC_RAM[16][8192]` | SRAM RP2350 |

`PIN_ROM_BLK` (GPIO 1 ? Q4 BC817 ? 74HC04 ? цепь BLK) принудительно устанавливается в LOW для блокировки собственного ПЗУ Spectrum. Микросхема ПЗУ Spectrum видит снятый сигнал chip-enable и немедленно переходит в высокоимпедансный режим на D0–D7. Ядро 1 затем выставляет образ DivMMC на шину через 74HC245 U1 и автомат состояний PIO1.

#### Автоматическая Активация через Перехват M1

Ядро 1 проверяет каждый цикл шины MREQ+/RD на наличие сигнала `/M1` (GPIO 3 = `PIN_M1`). При утверждении M1 (выборка опкода) считанный адрес запускает переходы автомата состояний:

| Диапазон адресов | Действие |
|-----------------|----------|
| `0x0000`, `0x0008`, `0x0038`, `0x0066`, `0x04C6`, `0x0562` (векторы RST) | Установить `divMMC_SW = true` — переключить `divMMC_ON` в конце цикла |
| `0x3D00–0x3DFF` (вход BDOS ESXDOS) | Принудительно `divMMC_ON = true` |
| `0x1FF8–0x1FFF` (трамплин выхода ESXDOS) | Установить `divMMC_SW = true` — переключить `divMMC_ON` в конце цикла |

Задержанное переключение `divMMC_SW` гарантирует, что текущий байт опкода будет полностью обслужен до смены окна памяти, предотвращая выборку Spectrum CPU устаревшего значения.

#### Взаимодействие NMI и ROM\_BLK с Аппаратурой

Когда DivMMC активен, линия `/NMI` удерживается низкой через GPIO 35 ? BC817 Q3 ? подтяжка R30 (300 Ом) к +5 В ? NemoBus `/NMI`. Это генерирует аппаратное NMI, заставляя ПЗУ ESXDOS перехватывать вызовы DOS. При деактивации DivMMC GPIO 35 переключается обратно на вход (высокий импеданс), Q3 выключается и линия NMI освобождается:

```c
if (divMMC_ON || divMMC_ON_PORT) {
    gpio_put(PIN_NMI, 1);
    gpio_set_dir(PIN_NMI, GPIO_OUT);  // Q3 включён ? /NMI утверждён LOW
} else {
    gpio_set_dir(PIN_NMI, GPIO_IN);   // Q3 выключён ? /NMI освобождён
}
```

### Клавиатура и Джойстик

#### Клавиатура PS/2 — PIO0 + Двойной DMA + Разъём Mini-DIN-6 (J6)

Физическая клавиатура PS/2 подключается к **J6** — 6-контактному разъёму Mini-DIN на плате. Защита по схеме:

- R12 (1 кОм) последовательно с CLK ? GPIO 38 (`PIN_PS2_CLK`)
- R13 (1 кОм) последовательно с DATA ? GPIO 39 (`PIN_PS2_DATA`)

Последовательные резисторы 1 кОм ограничивают ток защитных диодов RP2350 примерно до 1,2 мА, когда устройство PS/2 с питанием 5 В управляет входом GPIO 3,3 В, сохраняя безопасность RP2350B и обеспечивая корректные логические уровни (линии с открытым коллектором PS/2 подтянуты через собственные резисторы 10 кОм клавиатуры, что даёт устойчивый входной уровень GPIO близко к 3,3 В).

**Автомат состояний приёма PIO0 (`pio_program_PS2`):**

5-инструкционная PIO-программа непрерывно принимает 11-битные кадры PS/2 (старт + 8 бит данных + чётность + стоп):

```
set  x, 10          ; отсчитать 11 бит
wait 0 gpio, CLK    ; спад CLK PS/2 (вывод JMP = GPIO 38)
in   pins, 1        ; семплировать вывод DATA (GPIO 39)
wait 1 gpio, CLK    ; нарастание CLK
jmp  x--, 1        ; повторить для всех 11 бит
```

Делитель тактовой частоты PIO установлен в `clock_get_hz(clk_sys) / 200000` — примерно **200 кГц**, что обеспечивает надёжную выборку фронтов CLK PS/2 (10–16 кГц) даже при системной тактовой 351 МГц. Сдвиговый регистр приёма настроен на 11-битный ввод справа с автоматическим push в RX FIFO.

**Кольцевой буфер DMA без участия CPU:**

`init_PS2()` настраивает два цепочечных DMA-канала:

- **dma\_chan0**: RX FIFO PIO0 ? `PS_2_BUS[100]`, пакет 100 слов, темп задаётся DREQ PIO0, по завершении передаётся управление dma\_chan1.
- **dma\_chan1**: Записывает `&PS_2_BUS[0]` обратно в регистр адреса записи dma\_chan0, затем передаётся управление обратно dma\_chan0.

Результат: кадры PS/2 поступают незаметно в 100-словный кольцевой буфер без каких-либо затрат CPU на обоих ядрах.

**Декодирование сканкодов:**

`decode_PS2()` выполняется на **Ядре 0** каждые ~576 мкс (каждый 3-й тик ? каждую 64-ю итерацию). Он извлекает байты из кольцевого буфера, проверяет чётность, обрабатывает расширенные префиксы `0xE0`/`0xE1` и вызывает `translate_scancode()` для получения битовой маски `kb_state_t`.

**Трансляция матрицы клавиш ZX 8?5 (`zx_kb.c`):**

`set_zx_kb_state()` отображает каждый код `KEY_*` на один или несколько битов в массиве `zx_kb_state.a[8]`, точно моделирующем матрицу клавиатуры ZX Spectrum 8?5. Многоклавишные последовательности обрабатывают знаки препинания (например, `KEY_PERIOD` ? CAPS SHIFT + `N`).

**Ответ на шине со стороны Ядра 1:**

На любой цикл IORQ+/RD с `addr16 & 1 == 0` (порт клавиатуры Spectrum) отвечают декодированием 8 старших бит адреса как выборов строк:

```c
uint8_t addrh = addr16 >> 8;
uint8_t d8 = 0;
if ((addrh & 0x01) == 0) d8 |= kb_data[0];  // строка 0: CAPS SHIFT / Z / X / C / V
if ((addrh & 0x02) == 0) d8 |= kb_data[1];  // строка 1: A / S / D / F / G
...
put_dataZ80(~d8);  // активный низкий уровень, инвертируется перед выставлением на GPIO 24–31
```

Специальные комбинации: **Ctrl+Alt+Del** ? `reset_spectrum()`; **Insert** ? `NMI_press()` (управляет Q3, утверждает /NMI на 50 мс); **F12** ? принудительная перезагрузка через watchdog.

#### Джойстик Kempston — PCF8575DBR (U3) через I?C0 ? J10 (DE-9)

Физический джойстик подключается к **J10** — 9-контактному разъёму DE-9, соответствующему классической распиновке Atari/Kempston. Каждый вывод направления и кнопки огня подключается через подтягивающий резистор 10 кОм (R20–R28) к соответствующему P-порту **PCF8575DBR U3**:

| Вывод J10 | Сигнал | Вывод PCF8575 | Бит Kempston |
|-----------|--------|---------------|--------------|
| 1 | UP | P0 | бит 3 |
| 2 | DOWN | P1 | бит 2 |
| 3 | LEFT | P2 | бит 1 |
| 4 | RIGHT | P3 | бит 0 |
| 6 | FIRE | P4 | бит 4 |

PCF8575 доступен через **I?C0** на 400 кГц (GPIO 32 = SDA, GPIO 33 = SCL, адрес устройства `0x20`, A0=A1=A2=GND). Прошивка использует собственную неблокирующую I?C-функцию `i2c_transfer_non_blocking()`, которая:
1. Читает предыдущий результат I?C из аппаратного RX FIFO (результат предыдущего опроса).
2. Выдаёт следующую команду чтения в TX FIFO с битом STOP.

Функция возвращает управление немедленно, никогда не блокируя аудиотик Ядра 0 на 9 мкс. Сырое 16-битное показание PCF8575 преобразуется в формат Kempston через предварительно вычисленную таблицу из 256 элементов:

```c
// Пример режима NES (NES_JOY_EN=1):
data_kmpst = (nes_keys & 0x0F) | ((nes_keys >> 2) & 0x30)
           | ((nes_keys << 3) & 0x80) | ((nes_keys << 1) & 0x40);
data_kmpst ^= 0xFF;   // активный низкий ? активный высокий полярность Kempston
```

Результирующий байт `joy_state` обслуживается Ядром 1 при чтении порта `0x1F`. `kempston_port_enable` сбрасывается, пока CPU Spectrum выбирает из области ПЗУ (адрес < `0x3FFF`), чтобы избежать конфликтов с программами Spectrum, использующими этот диапазон адресов для данных ПЗУ.

### Распределение Задач По Ядрам

| Ядро | Функция |
|------|---------|
| **Ядро 0** | Синтез звука (TS, GS, микшер ? TDA1387T), эмуляция Z80 GS, декодирование PS/2, опрос джойстика I?C, watchdog |

На горячем пути не используется FIFO или спинлок — полагаясь на атомарность однобайтовых записей на ARM Cortex-M33.

### Ядро 0 — Тактируемый Аудиоцикл (период 9 мкс)

Ядро 0 выполняет цикл с жёстким таймером (`int_tick += 9`). На каждом тике:

1. `mix_sound_out()` — суммирует вклады GS, TS, Beeper, COVOX, SND\_DRIVE в 32-битный аккумулятор, применяет DC-блокиратор и мягкий ограничитель, затем вызывает `i2s_out()` для обновления `i2s_data`.
2. `ts.step()` — продвигает оба AY-чипа на один шаг синтеза.
3. Каждый 3-й тик: сканирование DMA-буфера клавиатуры PS/2 (`decode_PS2()`) и опрос джойстика I?C (`joy_proc()`).
4. Каждый 3-й тик (другая фаза): инъекция IRQ в Z80 GS (`IntZ80`) и `check_div_status()`.

Между тиками Ядро 0 непрерывно выполняет инструкции Z80 GS:

```c
int32_t dt = int_tick - tick_time;
ExecZ80(&cpu, dt * 12);  // ~12 тактов Z80 на мкс при эмулируемой частоте
```

Тик в 9 мкс даёт частоту обновления звука **?111 кГц**. TDA1387T потребляет семплы на частоте 44,1 кГц (определяется делителем тактовой PIO2), поэтому DMA считывает каждое значение `i2s_data` приблизительно 2,5 раза до его обновления — подход «ближайшего семпла» вместо кольцевого буфера.

### Ядро 1 — Монитор Шины ZX (цикл опроса)

`ZX_bus_task()` запускается на Ядре 1 через `multicore_launch_core1`. Выполняет бесконечный цикл `for(;;)`, считывая `gpio_get_all()` на каждой итерации:

1. Если `IORQ` и `MREQ` оба неактивны ? `is_new_cmd = true`, продолжить.
2. Если `RD` и `WR` оба неактивны ? неверный цикл, продолжить.
3. Четыре управляющих бита (`MREQ`, `IORQ`, `RD`, `WR`) инвертируются и используются как переключатель `switch` для четырёх типов цикла шины.
4. Флаг `is_new_cmd` гарантирует однократную обработку каждого цикла шины; сбрасывается при возврате всех управляющих линий в состояние покоя.

### Драйвер Шины PIO1 — Безглюковый Вывод Данных

Когда Ядро 1 вызывает `put_dataZ80(data)`:

```c
PIO_data_z80->txf[sm_data_z80] = 0x00FFFFFF | (data << 24);
```

Автомат состояний PIO1 (`pio_dataz80_program`) обрабатывает всё автономно на частоте 351 МГц (одна инструкция ? 2,8 нс):

```
0: mov pindirs, null   side 1  ? DIR=1 (B?A), GPIO24-31 = входы; DATA_CTRL=HIGH, Q1 выкл., U1 отключён
1: pull block          side 1  ? ожидание слова в TX FIFO
2: jmp pin, 0          side 1  ? /RD всё ещё HIGH (чтения нет)? отбросить, перезапустить
3: out pins, 8         side 1  ? загрузить байт в регистр выхода GPIO (пины ещё входы)
4: mov pindirs, ~null  side 0  ? DATA_CTRL=LOW ? Q1 вкл. ? U1 активен; GPIO24-31 = выходы (байт на шине)
5: jmp pin, 0          side 0  ? следить за /RD; при возврате HIGH ? вернуться к шагу 0
6: jmp 5               side 0  ? удерживать шину активной пока /RD низкий
```

Шаги 3?4 — критическая атомарная пара: байт предзагружается в регистр выхода, пока пины ещё входы, затем `pindirs` переключается в единственном PIO-цикле 2,8 нс с одновременным утверждением `DATA_CTRL` через side-set. Нет временного окна, в котором на D0–D7 может появиться некорректное значение.

### Путь Звука I?S — DMA-Управляемый, без Участия CPU

Программа `audio_i2s_program` (8 инструкций, `pio2`) реализует протокол I?S целиком в PIO:

- **Вывод данных**: GPIO 44; **BCLK**: GPIO 45; **LRCK**: GPIO 46.
- **Сдвиговый регистр**: 32-битный автоматический pull (выравнивание влево, MSB первым), с отображением на 16-битные семплы L + 16-битные семплы R.
- Биты `side-set` управляют BCLK и LRCK одновременно с каждым битом данных.
- **Делитель тактовой** вычисляется как `sys_clock / (44100 ? 8 ? 4)` для достижения точно 44 100 Гц независимо от разгона CPU.

После обновления `i2s_data` через `i2s_out(L, R)`:

```c
inline void i2s_out(int16_t l, int16_t r) {
    i2s_data = (((uint16_t)l) << 16) | ((uint16_t)r);
}
```

Два цепочечных DMA-канала (`dma_i2s` + `dma_i2s_ctrl`) наполняют TX FIFO PIO2:

- **dma\_i2s**: Читает `i2s_data` (без инкремента адреса чтения) ? FIFO PIO2, 1024 слова, темп задаётся DREQ PIO2 на 44,1 кГц. По завершении передаётся управление `dma_i2s_ctrl`.
- **dma\_i2s\_ctrl**: Записывает `1024` обратно в `dma_i2s.transfer_count` и передаёт управление обратно `dma_i2s`, перезапуская пакет.

Итог: TDA1387T получает непрерывный, свободный от джиттера поток I?S, управляемый исключительно DMA + PIO. Даже если Ядро 0 временно задержано долгим вызовом `ExecZ80()`, пара DMA будет просто продолжать отправлять последнее значение `i2s_data`, производя артефакт удерживаемого семпла (кратковременный тональный артефакт) вместо резкого щелчка.

### Аудиомикширование и Защита

`mix_sound_out()` суммирует знаковые 16-битные вклады от GS, TS, Beeper (`port & 0x0000` биты 3/4), COVOX (`port 0xFB`) и SND_DRIVE (`ports 0x0F/0x1F/0x4F/0x5F`) в 32-битный аккумулятор для каждого канала.

**DC-блокиратор** отслеживает долгосрочное среднее с помощью ФНЧ первого порядка (±1 на семпл):

```c
snd_L_C += SIGN_F(snd_L_in - snd_L_C);  // ±1 на семпл
snd_L_out = snd_L_in - snd_L_C;
```

**Мягкий ограничитель** (кусочно-линейный компрессор) предотвращает жёсткое ограничение.
Ниже `P_LIN = 20000`: линейное прохождение.  
Выше `P_LIN`: компрессионная кривая, стремящаяся к `P_MAX = 32760`:

$$
y = \frac{x \cdot P_{MAX} - P_{LIN}^2}{x - 2 \cdot P_{LIN} + P_{MAX}}
$$

## Потенциальные Слабые Места, Ошибки и Узкие Места

### 1. Задержка Джойстика I?C через PCF8575 — Входной Лаг и Зависание Bit-Bang NES

**Проблема:**

PCF8575 (U3) опрашивается через I?C0 на частоте 400 кГц. Полная транзакция чтения I?C (старт + байт адреса + ACK + 2 байта данных + STOP) требует минимум **~45 мкс** времени на шине при 400 кГц. В текущем пути `NES_MODE` функция `joy_proc()` выполняет многошаговую последовательность bit-bang через порт PCF8575 — утверждение LATCH, переключение CLK 8 раз, чтение данных — где каждый шаг требует отдельной записи I?C (обновление выходов порта) и последующего чтения I?C (семплирование DATA). Для полного цикла сдвига NES это составляет минимум **8 ? 2 ? 45 мкс = 720 мкс**, распределённых по нескольким вызовам `joy_proc()`.

Функция `i2c_transfer_non_blocking()` смягчает прямое зависание конвейеризацией: она выдаёт следующую транзакцию в аппаратный FIFO и читает предыдущий результат в одном вызове. Однако:

- **Частота опроса**: `joy_proc()` вызывается только каждый 3-й ? 9 мкс тик = каждые **27 мкс**. При такой частоте полный опрос NES занимает ? 27 мкс ? 20+ шагов ? **540+ мкс** туда-обратно.
- **Проверка переполнения FIFO I?C**: `i2c_transfer_non_blocking()` проверяет `i2c_get_write_available(i2c) < 16` перед выдачей следующей команды. Если предыдущая транзакция ещё не завершена (шина I?C занята предыдущим вызовом), функция выходит и возвращает `false`, молча пропуская этот шаг опроса. Нет ни повтора, ни индикации ошибки для вызывающей стороны.
- **Гранулярность обновления Kempston**: `joy_state` обновляется в конце последовательности NES (case 20 в `joy_proc()`). Если игровой цикл Spectrum читает порт `0x1F` между опросами NES, он видит устаревшее значение. При частоте кадров 50 Гц (20 мс) цикл опроса NES должен завершиться за ? 20 мс — он это делает — но игры с быстрым опросом (например, те, что крутятся на `IN 0x1F` в ожидании смены направления) могут наблюдать до **1 мс дополнительной задержки** между нажатием физической кнопки и чтением обновлённого значения Kempston Spectrum-ом.

**Симптомы:**
- Слегка вялая реакция джойстика в быстрых играх.
- Периодические пропуски однокадровых нажатий в ритмических или рефлекторных играх.
- В режиме NES: при перегрузке I?C `joy_state` может зависнуть на последнем полностью завершённом значении на несколько кадров.

**Рекомендуемые Исправления:**

1. **Использовать выделенный GPIO для каждого направления джойстика**, полностью исключив I?C для наиболее критичных по задержке линий. У RP2350B достаточно свободных GPIO — Kempston требует всего 5 бит.
2. **Для режима NES**: использовать PIO RP2350B для bit-bang протокола NES latch/clock нативно (идентично пути `DENDY_JOY_3_PINS`, который уже существует в кодовой базе, компилируется при `DENDY_JOY_3_PINS=1`). Это полностью устранило бы задержку I?C и сократило время полного опроса до нескольких тактов PIO.
3. **Для параллельного режима PCF8575** (Sega/Atari): перейти с I?C0 на пакетный I?C с опросом (оба байта в одной транзакции). Предварительно вычислить полную последовательность команд I?C как DMA-передачу, запускаемую повторяющимся таймером, записывая результаты в буфер памяти, из которого `get_joy_data()` читает без блокировки.
4. **Двойная буферизация `joy_state`**: использовать `__atomic_store_n(&joy_state, value, __ATOMIC_RELAXED)` и `__atomic_load_n` для предотвращения чтения Ядром 1 частично обновлённого байта (сейчас безопасно для `uint8_t`, но явные атомарные операции документируют намерение).

### 2. Конфликты на Шине и Гонки Сдвига Уровней

**Проблема A — Задержка распространения транзистора в цепи DATA\_CTRL:**

GPIO 2 (`PIN_DATA_CTRL`) управляет направлением и разрешением выхода 74HC245 U1 через транзистор BC817 Q1. Транзистор NPN вносит **задержку включения** примерно 5–15 нс (время хранения заряда базы при малом токе базы, задаваемом R23/R27 = 10 кОм). Это означает, что после утверждения `DATA_CTRL` PIO1 (шаг 4 программы PIO) существует окно, в котором 74HC245 ещё не полностью активировал свой выход:

```
PIO записывает pindirs (~null)  ? GPIO24-31 управляются на 3,3 В   [t=0, мгновенно]
PIO side-set DATA_CTRL=LOW      ? напряжение на базе Q1 начинает падать [t=~0,5 нс]
Коллектор Q1 достигает Vce(нас) ? CE/DIR U1 активны                [t=~10 нс]
```

В течение этого ~10 нс окна GPIO 24–31 уже управляют своими выходными значениями, но выходы 74HC245 (сторона B, 5 В) всё ещё в третьем состоянии. Строб /RD Spectrum может уже быть активным, и CPU Spectrum увидит неопределённые данные на D0–D7.

**Проблема B — Короткие окна /RD на быстрых клонах Spectrum:**

На стандартном ZX Spectrum 3,5 МГц строб /RD активен примерно **140–200 нс** после установки адреса. Цикл опроса Ядра 1 должен обнаружить /RD, отправить байт в FIFO PIO1 и получить от PIO1 управление шиной — всё в рамках этого окна. При 351 МГц Ядро 1 может выполнить ~3 инструкции за наносекунду. Плотный цикл опроса даёт приблизительно 5–8 итераций цикла в нс, что при наихудшем сценарии даёт задержку ответа от утверждения /RD до вызова `put_dataZ80()`:

```
наихудший случай = время завершения текущего обработчика case на Ядре 1
                 + накладные расходы вызова put_dataZ80
                 + запись в FIFO PIO + запуск PIO
```

Для простого чтения IORQ (без доступа к PSRAM) это комфортно укладывается в окно /RD. Однако:

- **Чтение MREQ с PSRAM**: когда `RdZ80()` обращается к странице, поддержанной PSRAM (slot2 или slot3), контроллер QMI вносит **~8–25 нс дополнительной задержки** на доступ в зависимости от границы страницы и параметра `COOLDOWN`. Для циклов MREQ, где адрес отображается на PSRAM, Ядро 1 должно вызвать `put_dataZ80()` в том же окне /RD после выборки PSRAM. На клоне Spectrum 7 МГц или машине Timex с работой на 4 МГц окно /RD может быть всего **80–100 нс** — что очень жёстко.

**Проблема C — Гонка утверждения ROM\_BLK при активации DivMMC:**

При активации DivMMC в середине последовательности `gpio_put(PIN_ROM_BLK, 0)` утверждает линию BLK. Это проходит через Q4 (BC817) и буфер 74HC04, добавляя ещё **10–20 нс**. Если ПЗУ Spectrum одновременно управляет D0–D7 (потому что ещё не увидело снятого CE), возникает кратковременный конфликт шины между буферами выхода ПЗУ Spectrum (5 В КМОП) и данными 3,3 В RP2350B через 74HC245 в момент перехода.

**Симптомы:**
- **Глюки на шине данных**: Spectrum читает неверные байты из эмулируемого ПЗУ или портов I/O в окне распространения транзистора, вызывая случайные сбои или неверные ответы I/O.
- **Зависание шины**: на быстрых клонах Spectrum Ядро 1 может не успеть ответить в окне /RD, заставляя CPU Spectrum семплировать плавающие данные (ёмкость шины удерживает последнее корректное значение) и производя непредсказуемое поведение.
- **Артефакты активации DivMMC**: первый байт, выбранный после активации DivMMC, может быть повреждён, если переход ROM\_BLK вступает в гонку с управлением ПЗУ Spectrum выходами D0–D7.

**Рекомендуемые Исправления:**

1. **Заменить BC817 на более быстрый MOSFET или специализированный преобразователь уровней** (например, однозатворный буфер SN74AHCT1G125 с задержкой распространения менее 1 нс). Это устранит задержку хранения заряда транзистора в пути DATA\_CTRL.
2. **Добавить короткую задержку PIO перед включением pindirs**: вставить `nop` или `nop [1]` между инструкциями `out pins, 8` и `mov pindirs, ~null`. Это даст транзистору и 74HC245 время полностью активироваться до появления данных на стороне B, ценой 2–3 дополнительных нс. В текущей программе PIO уже есть закомментированные инструкции `nop`, свидетельствующие о том, что это рассматривалось:
   ```
   // 0xa442, // 7: nop  side 0 [4]
   ```
3. **Проверить тайминги PSRAM при тактовой 7 МГц**: для совместимости с клонами Timex/Pentagon 7 МГц профилировать наихудшую задержку доступа к PSRAM и убедиться, что суммарное время ответа Ядра 1 (включая PSRAM) остаётся ниже 100 нс.
4. **Предотвратить конфликт шины ПЗУ Spectrum при DivMMC**: утверждать ROM\_BLK на один полный цикл CPU раньше, чем ожидается активация окна DivMMC. Текущая логика перехвата M1 может предварительно утверждать BLK при выборке `0x3CFF` (один адрес до точки перехвата `0x3D00`), гарантируя молчание ПЗУ Spectrum до прихода данных DivMMC.

### 3. Артефакт «Удержания Семпла» I?S — Функциональный Эквивалент Underrun

**Проблема:**

DMA-система I?S использует **единственную статическую переменную `uint32_t i2s_data`** в качестве источника чтения (нет кольцевого буфера). DMA настроен с `channel_config_set_read_increment(&cfg_dma, false)`, то есть читает с одного адреса (`&i2s_data`) при каждой передаче. Сигнал DREQ PIO2 задаёт темп этих передач точно **44 100 Гц**.

Ядро 0 вызывает `mix_sound_out()` ? `i2s_out(L, R)` с частотой **?111 111 Гц** (каждые 9 мкс), что в 2,52? быстрее скорости потребления ЦАПом. Это означает:

- Между двумя последовательными чтениями DMA Ядро 0 обновляет `i2s_data` приблизительно **2–3 раза**.
- Промежуточные вычисленные семплы **перезаписываются до попадания в ЦАП**, что делает эффективное аудиоразрешение ~44 100 Гц несмотря на 111 кГц частоту синтеза.
- Это намеренно и безвредно в нормальных условиях.

Однако если Ядро 0 **задержано более чем на ~22 мкс** (один период семпла ЦАПа при 44 100 Гц), DMA повторно прочитает предыдущее значение `i2s_data` и снова тактирует его в TDA1387T — производя артефакт удержания семпла. Источники таких задержек на Ядре 0:

1. **Длинные пакеты `ExecZ80()`**: вызов `ExecZ80(&cpu, dt * 12)` делается с `dt` = оставшиеся мкс до следующего аудиотика. При большом `dt` (например, после старта системы или долгого простоя) выполняются сотни тактов Z80 за один вызов. Каждая инструкция Z80, обращающаяся к PSRAM, добавляет ~8–25 нс, и длинный пакет с множеством обращений к PSRAM (например, при воспроизведении GS audio DMA) мог бы кратковременно увеличить суммарное время `ExecZ80` выше безопасного порога.
2. **Путь обновления watchdog**: `watchdog_update()` вызывается внутри цикла Ядра 0, но не внутри `ExecZ80()`. Если `ExecZ80()` выполняется дольше 500 мс (тайм-аут watchdog), watchdog срабатывает — сбрасывая всю систему. Это маловероятно, но теоретически возможно с патологичным ПЗУ GS.
3. **`check_div_status()` и обнаружение перемычки SD**: эта функция может вызвать `reset_spectrum()`, выполняющую `busy_wait_ms(10?50 = 500 мс)`. В течение этого времени Ядро 0 заблокировано, и DMA I?S просто повторяет последний семпл 500 мс — производя слышимую «застрявшую ноту».

**Симптомы:**
- Кратковременные тональные повторения или «заикания» при интенсивном доступе GS ROM к PSRAM (например, при загрузке большого банка семплов).
- Громкий **продолжительный тон** на 500 мс при каждом изменении перемычки DivMMC/ZC-SD (вызов `reset_spectrum()`).
- Возможный сброс watchdog при тяжёлых рабочих нагрузках ПЗУ GS.

**Рекомендуемые Исправления:**

1. **Заменить единственное слово `i2s_data` небольшим кольцевым буфером** (например, 8 или 16 стереосемплов). DMA будет использовать канал с инкрементом чтения, оборачивающийся в конце буфера, а Ядро 0 будет записывать в голову. Это разъединит частоту синтеза от частоты DMA и устранит артефакты удержания семпла при задержках до `buffer_size / 44100` секунд.

   Пример изменения конфигурации DMA:
   ```c
   // Заменить:
   channel_config_set_read_increment(&cfg_dma, false);
   // На:
   channel_config_set_read_increment(&cfg_dma, true);
   channel_config_set_ring(&cfg_dma, false, 5);  // 2^5 = 32 байта = кольцо 8 семплов
   ```

2. **Ограничить длину пакета `ExecZ80()`**: добавить верхнюю границу для `dt` перед вызовом `ExecZ80`, чтобы предотвратить неожиданно длинные единичные пакеты:
   ```c
   int32_t dt = int_tick - tick_time;
   if (dt > 100) dt = 100;  // ограничить 100 мкс = максимум 1200 тактов Z80 за пакет
   ExecZ80(&cpu, dt * 12);
   ```

3. **Вызывать `watchdog_update()` внутри `ExecZ80()` через регулярные интервалы**, реализовав проверку счётчика шагов внутри `LoopZ80()` (вызывается Z80-ядром между инструкциями). В настоящее время `LoopZ80()` просто возвращает `INT_NONE`; он мог бы вызывать `watchdog_update()` каждые N тактов.

4. **Заглушить звук во время `reset_spectrum()`**: перед вызовами `busy_wait_ms` записать в `i2s_data` тишину:
   ```c
   static void reset_spectrum() {
       i2s_out(0, 0);  // записать тишину перед блокировкой
       ROM_UNLOCK;
       ...
   }
   ```

### 4. Разделяемые volatile-Переменные — Отсутствующие Барьеры Памяти Между Ядрами

**Проблема:**

Несколько ключевых переменных разделяются между Ядром 0 (коллбэки Z80 GS, аудиомикшер) и Ядром 1 (цикл опроса шины) без явных инструкций барьера памяти:

```c
volatile uint8_t GSDAT, GSCOM, GSSTAT, ZXDATWR;
static sound_LR snd_out_GS, snd_out_TS;  // НЕ volatile
```

На **одноядерном** процессоре одного `volatile` достаточно — он предотвращает кэширование значения компилятором в регистре. На **двухъядерном Cortex-M33** в RP2350 каждое ядро имеет собственный конвейер, и **буферизация записи** может задержать видимость записи Ядра 1 для Ядра 0 и наоборот. Хотя ARM гарантирует атомарность однобайтовых выровненных записей, Cortex-M33 (в отличие от более старых M0/M0+) включает **буфер записи**, который может переупорядочивать сохранения. Без инструкции `__DMB()` (Data Memory Barrier) последовательность записей в разделяемые переменные может наблюдаться другим ядром в другом порядке.

Конкретно:

```c
// Ядро 1 записывает:
GSDAT = d8;         // сохранение 1
GSSTAT |= DataBIT;  // сохранение 2 (чтение-модификация-запись volatile uint8_t)

// Ядро 0 читает:
if (GSSTAT & DataBIT)  // может наблюдать DataBIT установленным до нового значения GSDAT
    process(GSDAT);    // читает устаревший GSDAT
```

Строка `GSSTAT |= DataBIT` — это операция чтения-изменения-записи, компилируемая в последовательность LOAD + ORR + STORE. STORE `GSSTAT` может наблюдаться Ядром 0 до STORE `GSDAT`, если буфер записи опустошается непоследовательно.

Кроме того, `snd_out_GS` и `snd_out_TS` (`sound_LR`, два `int16_t`) **не объявлены `volatile`**. Ядро 0 записывает их; Ядро 1 их не читает — но если компилятор решит оптимизировать одну или обе записи в регистр через границу тика, аудиомикшер может прочитать устаревшие значения.

**Симптомы:**
- Редкие сбои коммуникации GS: Z80 GS видит установленный `GSSTAT.DataBIT`, но читает `GSDAT = 0x00` (старое значение), заставляя обрабатывать нулевой байт данных вместо реальной команды.
- Очень редко — повреждённые пары `GSCOM`/`GSDAT`, вызывающие зависание прошивки GS или воспроизведение неверного звука.
- Эти ошибки **недетерминированы** и могут не воспроизводиться на осциллографе — они проявляются как случайные аудиоглюки или зависания GS при интенсивном использовании.

**Рекомендуемые Исправления:**

1. **Добавить `__DMB()` после записей в разделяемые пары флаг+данные:**
   ```c
   GSDAT = d8;
   __DMB();            // гарантировать видимость GSDAT до обновления GSSTAT
   GSSTAT |= DataBIT;
   ```
   И симметрично на Ядре 0:
   ```c
   uint8_t dat = GSDAT;
   __DMB();
   GSSTAT &= ~DataBIT;
   ```

2. **Использовать примитивы Pico SDK `__sev()`/`__wfe()` или `multicore_fifo_push_blocking()`** для пути почтового ящика команда/данные. Для низкочастотной передачи команд (команды GS поступают на человеческих скоростях) накладные расходы FIFO пренебрежимо малы, а корректность гарантирована.

3. **Объявить `snd_out_GS` и `snd_out_TS` `volatile`** для предотвращения откладывания записей компилятором через границу тика.

### 5. Подключение Двух SD-Разъёмов — Скрытый Конфликт на Шине

**Проблема:**

J7 (6-контактный разъём) и J\_SD1 (разъём Micro-SD) подключены **параллельно** к одним линиям шины SPI1 (GPIO 40/42/43) и одной линии CS0 (GPIO 41). Отдельного chip-select для индивидуального управления двумя картами нет. Схема не предусматривает аппаратной защиты от одновременной вставки карт в оба разъёма.

При **одновременной вставке двух SD-карт**:

- Оба карты будут иметь активные выходы MISO одновременно. Стандартные SD-карты не переводят свой MISO в третье состояние при снятом CS (это делается в режиме SPI только если CMD0 был правильно отправлен с индивидуальным CS). При включении питания, до инициализации карт, обе карты могут одновременно управлять конфликтующими логическими уровнями на GPIO 40 (SD\_MISO), вызывая конфликт шины с коротким замыканием на линии MISO 3,3 В.
- GPIO 40 защищён только внутренними защитными диодами периферии SPI1 без последовательного резистора. Продолжительный конфликт шины может повредить контакт GPIO.

**Симптомы:**
- Сбой инициализации SD-карты или загрузки ESXDOS при наличии двух карт.
- Захват GPIO 40 на среднем напряжении (~1,6 В), что заставляет данные SPI выглядеть как `0xFF` или `0x00` при всех чтениях.
- В крайних случаях — повреждение контакта GPIO 40, вызывающее постоянный сбой RX SPI1.

**Рекомендуемые Исправления:**

1. **Добавить последовательный резистор 33–47 Ом** на линии MISO каждой SD-карты до их объединения в общую цепь. Это ограничивает ток короткого замыкания до ~100 мА даже в наихудшем случае и предотвращает повреждение GPIO. Резисторы также улучшают целостность сигнала, демпфируя отражения на цепи MISO.

2. **Использовать отдельные линии CS для каждого разъёма** (выделить один свободный GPIO, например GPIO 31, для CS J7). Обновить прошивку для использования правильного CS в зависимости от активной карты, и задокументировать, что одновременно должна быть установлена только одна карта.

3. **Добавить аппаратный вентиль OR или диодную схему И на MISO** (два диода 1N4148 с катодами к общей подтяжке), чтобы невыбранная карта не управляла линией MISO.

### 6. Ошибка Вычисления Частоты PSRAM в `psram_init()`

**Проблема:**

Функция `psram_init()` вычисляет делитель QMI используя **жёстко закодированную константу 400 МГц** вместо вызова `clock_get_hz(clk_sys)` для запроса реальной рабочей частоты:

```c
const int clock_hz = 400000000;  // ЖЁСТКО ЗАКОДИРОВАНО — не читает реальную системную частоту
int divisor = (clock_hz + max_psram_freq - 1) / max_psram_freq;
// При clock_hz=400M, max_psram_freq=166M: divisor = ceil(400/166) = 3
// Реальная системная частота = 351 МГц ? 351/3 = 117 МГц ? в рамках спецификации (OK в этом случае)
```

При текущей системной тактовой 351 МГц с делителем 3 частота PSRAM составляет 117 МГц — хорошо в пределах спецификации APS6404L (133 МГц максимум при 3,3 В или 166 МГц при VDD = 3,6 В). Однако:

- При изменении `CPU_FREQ` в `config.h` на значение между 332 МГц (где делитель 3 всё ещё даёт 110 МГц, безопасно) и 498 МГц (где делитель 3 даёт 166 МГц, на пределе), жёстко закодированное предположение о 400 МГц означает, что прошивка **недооценит делитель** для тактовых выше 400 МГц, потенциально устанавливая тактовую PSRAM выше номинальной частоты.
- `rxdelay` также вычисляется из `clock_hz = 400 МГц`, а не из реальной тактовой. Если реальная тактовая > 400 МГц, окно захвата считанных данных может сместиться, вызывая ошибки чтения QMI, которые проявляются как случайное повреждение памяти в банках GS RAM, поддержанных PSRAM.

**Симптомы:**
- Случайные сбои Z80 GS или аудиоглюки при изменении `CPU_FREQ` выше текущего значения 351 МГц.
- Повреждение данных в массивах `RAM_BANK1[n]`/`RAM_BANK2[n]`, проявляющееся как неверное воспроизведение звука или зависания прошивки GS.
- Ошибка **скрыта** при 351 МГц, но активируется при увеличении разгона.

**Рекомендуемое Исправление:**

Заменить жёстко закодированную константу вызовом `clock_get_hz()`. Комментарий в существующем коде даже признаёт это:

```c
// До (ошибочно):
const int clock_hz = 400000000;

// После (правильно):
const int clock_hz = clock_get_hz(clk_sys);
```

Это **однострочное исправление** без побочных эффектов при текущей рабочей точке 351 МГц.

### 7. Пробел в Декодировании Адресного Пространства GS ROM — Недостижимые Адреса Возвращают 0

**Проблема:**

В `RdZ80()` логика покрытия карты памяти имеет пробел. Рассматривая условия:

```c
if (address < 0x4000)  ? ПЗУ (slot0)
if (address > 0x3fff && address < 0x8000)  ? slot1 (фиксированная SRAM) или захват канала
if (address > 0x7fff && slot23_is_ram != 1) ? ПЗУ (slot0, маскировано на 0x7FFF)
else if (address > 0x7fff && address < 0xc000) ? slot2
else if (address > 0xbfff && address <= 0xffff) ? slot3
return 0;  ? ПО УМОЛЧАНИЮ
```

Логика корректна, но хрупка — цепочка `else if` зависит от точной структуры, и будущий рефакторинг с переупорядочиванием условий мог бы незаметно ввести пробел, возвращающий `0x00` для адресов `0x8000–0xFFFF` при активном банкировании.

Более глубокая проблема: когда `slot23_is_ram == 0` (режим ПЗУ) И `address` находится в `0x8000–0xFFFF`, код корректно возвращает `gs105b_rom[address & 0x7FFF]`. ПЗУ GS **всего 32 КБ** (`gs105b_rom[]` в `gs105b.rom.h`). Его зеркалирование по `0x8000–0xFFFF` — корректное поведение аппаратуры GS. Но если заголовок ПЗУ когда-либо изменит размер или выравнивание, это молчаливое зеркалирование может давать неверные данные.

**Симптомы:**
- Если будущий разработчик добавит клаузу `else` или переупорядочит условия `RdZ80`, Z80 GS может выполнять инструкции `0x00` (NOP) из неотображённых областей вместо ПЗУ, застревая в NOP-цепочке и прекращая воспроизведение звука.

**Рекомендуемое Исправление:**

Рефакторинг `RdZ80()` с использованием явного switch-case или таблицы диапазонов вместо цепочки `if`/`else if` с перекрывающимися условиями:

```c
byte RdZ80(word addr) {
    if (addr < 0x4000)
        return (slot0_is_ram) ? ram_slot0[addr] : gs105b_rom[addr];
    if (addr < 0x8000)
        return ram_slot1[addr & 0x3FFF];   // также вызывает побочный эффект захвата канала
    if (!slot23_is_ram)
        return gs105b_rom[addr & 0x7FFF];  // зеркало ПЗУ в верхних 32 КБ
    if (addr < 0xC000)
        return ram_slot2[addr & 0x3FFF];
    return ram_slot3[addr & 0x3FFF];
}
```

Это устраняет все перекрывающиеся условия и делает возврат по умолчанию явным для каждого адреса.
